# Core Concepts

The ECS pattern has four key primitives: **entities**, **components**, **systems**, and **bundles**. Understanding how they interact is essential to building with this framework.

## Entities

An entity is just an ID ā€” a lightweight identifier for a UI node. Entities don't hold data or behavior directly. Instead, data is attached to them via components.

<div data-code-tabs>

```tsx
<Entity>
  {/* Components and children go here */}
</Entity>
```

```ts
const entity = world.createEntity(undefined, [
  // SomeComponent(...),
]);
```

</div>

In the DOM, each entity with a `DOMElement` component renders as exactly one DOM node. This is a hard rule: **one `DOMElement` per entity**.

## Components

Components are typed data containers attached to entities. They hold state, configuration, or references ā€” but no logic.

```ts
import { defineComponent, defineMarker } from "@ecs-test/ecs";

// A component with data
const Position = defineComponent<{ x: number; y: number }>("Position");

// A marker component (no data, just a flag)
const Selected = defineMarker("Selected");
```

Common built-in components:

| Component | Purpose |
|-----------|---------|
| `DOMElement` | Renders a DOM element (`div`, `button`, etc.) |
| `Classes` | CSS class names |
| `Clickable` | Marks an entity as interactive (`Clicked` is set on click) |
| `Disabled` | Disables interaction |
| `TextContent` | Sets element text content |

### add vs set

- `world.add(entity, ComponentInstance)` ā€” initial setup, throws if component already exists
- `world.set(entity, ComponentInstance)` ā€” upsert, creates or updates

### Mutable Component Updates

Component data is mutable. Use the update primitive that matches the amount of control you need:

- `world.set(...)` when you are replacing the whole component value
- `world.mutate(entity, Component, data => { ... })` for the safe in-place update path
- `world.getMutable(...)` + `world.markMutated(...)` + `world.flush()` when you intentionally want full manual control

Example:

```ts
world.mutate(entity, SearchQuery, query => {
  query.value = "ecs";
});
```

## Systems

Systems are reactive functions that run when entities enter, exit, or change within a query. They contain all the logic.

```ts
import { defineReactiveSystem, Entities } from "@ecs-test/ecs";

const HighlightSystem = defineReactiveSystem({
  query: Entities.with([Selected, Classes]),
  onEnter(world, entities) {
    for (const entity of entities) {
      const classes = world.get(entity, Classes);
      if (!classes) continue;
      world.set(entity, Classes({ list: [...classes.list, "highlighted"] }));
    }
  },
  onExit(world, entities) {
    for (const entity of entities) {
      const classes = world.get(entity, Classes);
      if (!classes) continue;
      world.set(entity, Classes({ list: classes.list.filter((c) => c !== "highlighted") }));
    }
  },
});
```

Systems react to **query membership changes**:
- `onEnter` ā€” entity just matched the query (component added)
- `onExit` ā€” entity no longer matches (component removed)
- `onUpdate` ā€” a required component's data changed

### No Frame Loop

Unlike game ECS, there is no `requestAnimationFrame` loop. Systems run during flush cycles. By default, mutations auto-trigger a flush (`autoFlush: true`), and you can disable that for manual batching.

## Entity Events

For cross-entity UI intent wiring, define typed events and emit them through entity hierarchy:

```ts
import { defineEvent } from "@ecs-test/ecs";

const TodoToggle = defineEvent<{ todoId: number }>("todo.toggle");

world.on(todoRow, TodoToggle, (event) => {
  const { todoId } = event.payload;
  // update state...
  event.bubble();
});

world.emit(toggleButton, TodoToggle, { todoId: 42 });
// equivalent: world.emit(toggleButton, TodoToggle({ todoId: 42 }))
```

Event propagation walks from `sourceEntity` up through parents. A handler stops propagation by default, and can continue upward only by calling `event.bubble()`.

## Binding Targets

For DOM input/value synchronization, bindings target components with a `{ value: ... }` shape.

Typical binding paths look like:

```ts
binding(SearchQuery)
binding(ProfileState).fieldMaybe("nickname")
binding(TripState).field("travellers").by("id", travellerId).field("name")
binding(StepsState).field("steps").at(0)
```

Core rules:
- owner resolution is nearest matching component, including the current entity
- `.field(...)` descends into object data
- `.fieldMaybe(...)` allows legitimate optional leaf values
- `.at(i)` is positional array access
- `.by(key, value)` is semantic array-item lookup for repeated widgets

`DOMBinding` then connects a binding target to an actual DOM element:

```tsx
<Entity>
  <DOMElement tag="input" attrs={{ type: "search", placeholder: "Search docs" }} />
  <SearchQuery value="" />
  <DOMBinding
    bind={binding(SearchQuery)}
    readEvent="input"
    read={(el) => (el as HTMLInputElement).value}
    write={(el, value) => {
      (el as HTMLInputElement).value = String(value ?? "");
    }}
  />
</Entity>
```

## Bundles

Bundles group multiple components into a reusable unit. They're the primary composition tool.

```ts
import { defineBundle } from "@ecs-test/ecs";
import { DOMElement, Clickable, Classes, TextContent } from "@ecs-test/dom";

const Button = defineBundle(({ label }: { label: string }) => [
  DOMElement({ tag: "button" }),
  Classes({ list: ["btn"] }),
  TextContent({ value: label }),
  Clickable(),
]);
```

Use in JSX:

<div data-code-tabs>

```tsx
<Entity>
  <Button label="Click me" />
</Entity>
```

```ts
const button = world.createEntity(undefined, [
  DOMElement({ tag: "button" }),
  Classes({ list: ["btn"] }),
  TextContent({ value: "Click me" }),
  Clickable(),
]);
```

### Customizing Bundles

Use `except` to remove components from a bundle, or `only` to keep specific ones:

```tsx
{/* Button without click handling */}
<Entity>
  <Button label="Display only" except={[Clickable]} />
</Entity>
```

```ts
const displayOnly = world.createEntity(undefined, [
  DOMElement({ tag: "button" }),
  Classes({ list: ["btn"] }),
  TextContent({ value: "Display only" }),
]);
// Intentionally no Clickable component.
```

</div>

## The Flush Cycle

With `autoFlush: true` (default), each mutation flushes immediately. For batching workflows, set `autoFlush: false` and flush manually.

Manual flush flow:

1. Add/set/remove components (queued)
2. Call `world.flush()`
3. Query membership recalculated
4. Systems run (`onEnter`, `onUpdate`, `onExit`)
5. DOM updates applied by DOM systems

This batching prevents cascading re-renders and keeps execution predictable.

Mutable update note:
- prefer `world.mutate(entity, Component, data => { ... })` for in-place updates that should participate in reactivity
- when you need full manual control, use `world.getMutable(...)`, mutate the returned object, call `world.markMutated(entity, Component)`, then `world.flush()` when appropriate
- `world.batch(...)` remains the safer way to group multiple updates without having to manage per-mutation flush timing manually
