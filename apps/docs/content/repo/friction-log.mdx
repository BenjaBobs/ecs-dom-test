# Friction Log

This log captures framework friction discovered while building the docs app and the UI standard library.

Use this with:
- [Roadmap](/repo/roadmap.html) for milestone context
- [Working Set](/repo/working-set.html) for active execution state

## Categories

- `footgun`: easy-to-make mistakes with high cost
- `verbosity`: too much code for common patterns
- `inflexibility`: valid composition the framework currently resists
- `missing_feature`: capability needed but not present

## Active Findings

| ID | Date | Category | Finding | Milestone |
| --- | --- | --- | --- | --- |
| FR-20260214-04 | 2026-02-14 | verbosity | Interactive docs examples require too much boilerplate (`defineMarker`, query lookups, manual label systems) even for simple counters. Direction: higher-level primitives in `@ecs-test/ui`. | R3 |
| FR-20260322-03 | 2026-03-22 | missing_feature | No input/form component — reading an input value requires `getDOMElement` + casting to `HTMLInputElement`. Every form-like app hits this immediately. Direction: add an `InputValue` component that syncs to/from the DOM element. | R3 |
| FR-20260322-04 | 2026-03-22 | missing_feature | `Clicked` is a bare marker with no payload. Entity-per-item + `getParent()` works for hierarchical cases, but flat layouts still need extra data components to identify click targets. **Partially addressed**: `Clickable({ onClick })` eliminates marker-per-button boilerplate for self-contained actions, and typed entity events (`defineEvent` + `world.emit/on`) now provide payloaded intent routing with optional bubbling. Remaining gap: codified DOM feature integration patterns on top of the event primitives. | R3 |
| FR-20260322-05 | 2026-03-22 | verbosity | Building child entities for a dynamic row is verbose in non-JSX (~7 `world.add()` calls per sub-entity). JSX + `materialize` helps but is only available at setup time or via explicit calls inside systems. Direction: consider a list-rendering helper or batch entity builder. | R3 |
| FR-20260322-06 | 2026-03-22 | footgun | `onUpdate` only fires when a **query component** changes, not when any component on a matched entity changes. Adding `Done` to an entity with `[TodoText, DOMElement]` does NOT trigger `onUpdate` for a `[TodoText, DOMElement]` query. Workaround: use two systems with complementary queries (`with([Done])` / `without([Done])`). Direction: consider an `onChange` callback that fires for any component change on matched entities, or document the semantics more prominently. | R3 |
| FR-20260322-08 | 2026-03-22 | verbosity | Updating visuals after a marker change (e.g., `Done` toggled) currently requires tearing down all children and rebuilding. **Partially addressed**: child-scoped lookup APIs (`world.queryChildren`, `world.findChild`, `world.findChildMaybe`) now support targeted direct-child updates. Remaining gap: codified row/widget patterns that use these APIs for surgical updates by default. | R3 |
