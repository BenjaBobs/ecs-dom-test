# Current Todo

This page is the detailed design exploration for the work currently driving `R3` in the [Working Set](/repo/working-set.html). It is intentionally more exploratory than the roadmap or working-set pages.

Use this page for:
- the current design direction
- the reasoning behind design decisions
- alternatives we considered and rejected
- open questions that still need answers before or during implementation

## Problem Statement

We want the framework to be:
- flexible enough for complex composite widgets and nested widget trees
- simple enough that the base model is teachable without a large conceptual tax
- not overly verbose for common DOM and form-like flows

The starting pain points were:
- manual `getDOMElement(...)` reads and casts for basic input flows
- feature-specific DOM sync systems for cases like docs search
- direct DOM <-> form coupling in `forms-ui`
- unclear design story for composite widgets made from multiple entities

## Concrete Starting Point

The discussion started from patterns like this in docs code:

```tsx
<Entity>
  <DOMElement tag="input" />
  <Classes list={["search-input"]} />
  <SearchInput />
  <SearchQuery value="" />
</Entity>
```

paired with a feature-specific system:

```ts
const SearchInputSystem = defineReactiveSystem({
  name: "SearchInputSystem",
  query: Entities.with([SearchInput]),
  onEnter(world, entities) {
    for (const entity of entities) {
      const el = getDOMElement(world, entity) as HTMLInputElement | null;
      if (!el) continue;

      el.type = "search";
      el.placeholder = "Search docs";
      el.autocomplete = "off";
      el.spellcheck = false;

      const query = world.get(entity, SearchQuery);
      if (query) {
        el.value = query.value;
      }

      el.addEventListener("input", () => {
        world.set(entity, SearchQuery({ value: el.value }));
        world.flush();
      });
    }
  },
});
```

This is workable, but it repeats the same pattern many features will need:
- configure the element
- read a DOM property
- write a component value
- write the DOM property back later when authoritative state changes

That repetition is what started the `DOMBinding` exploration.

## What We Agreed On

### Keep `DOMElement` Clean

We considered merging binding behavior into `DOMElement` to improve TypeScript inference, especially for event names. We rejected that.

Why:
- `DOMElement` should stay a renderer primitive
- binding and rendering are separate concerns
- many entities need a `DOMElement` without any value synchronization at all
- merging binding into `DOMElement` would improve local typing but weaken the architecture

Current direction:
- `DOMElement` stays separate from binding
- `DOMElement` may carry static intrinsic `attrs`
- `attrs` should not absorb behavior, content, or reactive synchronization concerns

### `DOMElement.attrs` Is Useful, but Narrowly Scoped

We want examples like the docs search input to stop needing a small feature-specific system just to set:
- `type`
- `placeholder`
- `autocomplete`
- `spellcheck`

Current direction:
- `DOMElement` gets static `attrs`
- `attrs` is for intrinsic static element configuration only

Intended examples:
- `type`
- `placeholder`
- `autocomplete`
- `src`
- `href`
- `role`
- `tabIndex`
- `aria-*`
- `data-*`

Explicit exclusions:
- `class`
- `style`
- text or HTML content
- event handlers
- reactive `value` / `checked` synchronization

Why:
- `Classes`, `Style`, `TextContent`, and `HtmlContent` already provide better ECS-specific semantics
- keeping these separate preserves single sources of truth

### `DOMBinding` Should Stay Separate from `DOMElement`

We agreed `DOMBinding` is the right place for DOM <-> state synchronization.

Current direction:
- `DOMBinding` remains its own component/system
- `readEvent` is the right name for the DOM-to-state trigger
- read-side timing controls such as debouncing belong on `DOMBinding`
- writes back to the DOM should remain immediate by default

Why:
- keeps the base model explicit
- preserves room for consumers to build other binding systems if needed
- avoids forcing a large kitchen-sink API into `DOMElement`

### Avoid `__entity` / `__world` on DOM Nodes by Default

We considered storing ECS references directly on DOM nodes for lookup speed.

Current direction:
- do not attach `__entity` / `__world` by default

Why:
- current interaction systems already capture `world` and `entity` in closures
- the renderer already owns a world-scoped entity -> element map
- element-attached ECS references would complicate the renderer model
- revisit only if profiling later proves a real bottleneck

## How the Binding Design Evolved

### Step 1: `Value` + `DOMBinding`

The initial direction was:
- a generic `Value` component
- `DOMBinding` only targets `Value`
- higher-level abstractions like `InputValue` could be built on top

Why it seemed promising:
- very clear mental model
- clean layering
- no direct DOM <-> form coupling

Problem discovered:
- it makes queries broad when many entities have `Value`
- feature systems would need extra markers just to narrow those queries again
- domain-specific value components like `SearchQuery` would need awkward adapter systems

Rejected pseudo-code:

```tsx
<Entity>
  <DOMElement tag="input" />
  <Value value="" />
  <DOMBinding
    readEvent="input"
    read={el => (el as HTMLInputElement).value}
    write={(el, value) => {
      (el as HTMLInputElement).value = String(value ?? "");
    }}
  />
</Entity>
```

If the feature still wants a meaningful domain component such as `SearchQuery`, then another system is needed:

```ts
const SearchQuerySyncSystem = defineReactiveSystem({
  name: "SearchQuerySyncSystem",
  query: Entities.with([Value, SearchInput]),
  onEnter: sync,
  onUpdate: sync,
});

function sync(world, entities) {
  for (const entity of entities) {
    const value = world.get(entity, Value);
    if (!value) continue;
    world.set(entity, SearchQuery({ value: String(value.value ?? "") }));
  }
}
```

That extra sync step is what made the strict `Value`-only approach feel wrong.

### Step 2: `DOMBinding` Targets an Explicit Value-Shaped Component

We refined the design so `DOMBinding` points at a chosen component with a `{ value: ... }` shape.

Example:

```tsx
<SearchQuery value="" />
<DOMBinding
  component={SearchQuery}
  readEvent="input"
  read={el => (el as HTMLInputElement).value}
  write={(el, value) => {
    (el as HTMLInputElement).value = String(value ?? "");
  }}
/>
```

Why this is better than strict `Value`-only:
- keeps domain state meaningful
- avoids extra adapter systems
- keeps queries narrow
- still preserves a generic `Value` as the default catch-all where no domain component exists

Refined example:

```tsx
<Entity>
  <DOMElement
    tag="input"
    attrs={{
      type: "search",
      placeholder: "Search docs",
      autocomplete: "off",
      spellcheck: false,
    }}
  />
  <Classes list={["search-input"]} />
  <SearchQuery value="" />
  <DOMBinding
    component={SearchQuery}
    readEvent="input"
    read={el => (el as HTMLInputElement).value}
    write={(el, value) => {
      (el as HTMLInputElement).value = String(value ?? "");
    }}
  />
</Entity>
```

This version removes most of the feature-specific plumbing and keeps `SearchQuery` as the feature state.

### Step 3: Composite Widgets and Root-Owned State Forced a Bigger Rethink

Once we looked at widgets made from multiple entities, especially forms and widget-of-widgets cases, a same-entity-only component target was not enough.

Example pressure case:
- a date range widget made from label/input/error descendants
- authoritative value may belong to a form root or widget root, not the input entity itself

This forced the deeper question:
- should the real abstraction be component-to-component sync, or binding to authoritative state wherever it lives?

## Root-Owned Data Is the Real Underlying Problem

We concluded that `forms-ui` should be treated as a vertical slice proof-of-concept, not the foundational architecture.

What the current `forms-ui` package already proves:
- descendant entities often need to bind to data owned by an ancestor/root entity
- root data often has rich concerns layered on top: validation, touched, dirty, errors

What this implies:
- the reusable foundation is not “forms binding”
- it is “root-owned data binding”

This suggests the real base problem is:
- define root-owned data on some entity
- let descendant entities resolve the nearest relevant owner
- bind to some nested value in that root-owned data

Why this matters:
- forms are one important case, but not the only one
- composite widgets and nested widgets need the same pattern

The current `forms-ui` pattern already hints at this shape:

```tsx
<Entity>
  <FormData factory={TripForm} />

  <Entity>
    <DOMElement tag="input" />
    <TextInput />
    <FormBinding field={TripForm.fields.name} />
  </Entity>
</Entity>
```

The important architectural fact is not the form package itself. It is:
- the authoritative state lives on a root/ancestor
- descendants bind to pieces of that state

## Root Resolution: What We Rejected and What We Kept

### Generic `Root` Marker

We considered a generic `Root` marker and a walk-up-to-nearest-root function.

Why it looked attractive:
- simple
- reusable
- establishes clear subtree boundaries

Why we did not choose it as the primary data-binding lookup:
- a generic `Root` does not tell you what kind of state lives there
- nested roots would introduce ambiguity
- for data binding, “nearest root” is usually too broad

### Nearest Ancestor with Specific Component

We preferred:
- walk up to the nearest ancestor that has the specific component the binding wants
- include the current entity in the search

Why:
- more precise than a generic `Root`
- works naturally for widget-of-widgets
- outer and inner widgets can coexist cleanly
- nested scopes shadow each other by component type, which is usually exactly what we want

Conceptually:

```ts
function findNearestOwner(world, entity, component) {
  let current = entity;
  while (current != null) {
    if (world.has(current, component)) {
      return current;
    }
    current = world.getParent(current);
  }
  return undefined;
}
```

The important rule is that the current entity counts as the first candidate. So this works for:
- same-entity state
- nearest ancestor state
- nested widget roots

## The Widget-of-Widgets Story

We explored nested widgets such as:
- outer `FormState`
- inner `DateRangeState`
- descendant input entities for start/end values

Important realization:
- a widget bundle should not necessarily own copied authoritative value state
- if the authoritative value already exists in a root-owned data structure, the widget should bind directly to it

Why:
- copying introduces duplication
- mirroring through systems can introduce timing/schedule lag
- widget-heavy UIs may suffer performance and complexity costs from unnecessary sync hops

So the current direction is:
- authoritative domain value should usually be bound directly
- local widget state should only exist for transient UI concerns such as:
  - open/closed state
  - focus/selection state
  - draft text
  - temporary parse state

Illustrative anti-pattern:

```tsx
<DateRangeState value={{ start: "", end: "" }} />
```

plus a later sync into `FormState.range`.

Why this is risky:
- duplicate state
- extra synchronization logic
- potential schedule delay

Preferred direction:
- `FormState.range` is the authoritative value
- `DateRangeBundle` binds directly to `range.start` and `range.end`
- local widget state is only added if the widget has extra transient concerns

## Binding Targets

This is the current promising abstraction.

Instead of making `DOMBinding` know whether it is talking to:
- a local component
- an ancestor-owned data object
- a form field

we move that concern into a binding target.

Conceptually, a binding target answers:
- how to get the current value
- how to set the next value
- how to subscribe to authoritative changes

Then `DOMBinding` only needs to know:
- how to read from the DOM
- how to write to the DOM
- which binding target it should talk to

Conceptual minimal interface:

```ts
type BindingTarget<T> = {
  get(world: World, entity: EntityId): T;
  set(world: World, entity: EntityId, value: T): void;
  subscribe(world: World, entity: EntityId, notify: () => void): () => void;
};
```

This is intentionally broad. It can represent:
- same-entity component values
- nearest-ancestor values
- form field accessors
- future adapted state containers

### Why Binding Targets Are Attractive

- they unify local and root-owned state
- they work equally well for inline entities and bundles
- they avoid duplicated authoritative state
- they let forms become one specialization of a more general mechanism

## Fluent Binding Builder Direction

The current most promising user-facing API shape is fluent.

Conceptually:

```ts
binding(SearchQuery)
binding(DateRangeState).field("start")
binding(TripState).field("travellers").at(i).field("name")
```

The intended semantics:
- `binding(Component)` means “find the nearest entity with this component, including the current entity”
- `.field(...)` narrows into nested object data
- `.at(...)` narrows into array items

Why this is appealing:
- one mental model
- no hard split between same-entity and ancestor binding APIs
- feels appropriate for both inline code and bundle internals

Illustrative fluent usage:

```ts
binding(SearchQuery)
binding(DateRangeState).field("start")
binding(TripState).field("range").field("end")
binding(TripState).field("travellers").at(i).field("name")
```

Current intuition:
- `binding(Component)` resolves scope
- `.field(...)` and `.at(...)` navigate within the authoritative data inside that scope
- forms should ideally fit into the same fluent shape, even if they need an adapter internally

## Search Example: Before and After

The docs search input is a useful design test.

Current implementation:
- a `SearchInput` marker
- a feature-specific system that:
  - gets the DOM node
  - sets `type`, `placeholder`, and other element properties
  - attaches the `input` listener
  - writes to `SearchQuery`

Refined intended direction:
- move static input configuration into `DOMElement.attrs`
- bind the input directly to `SearchQuery`
- leave the feature system responsible only for search-specific behavior

Pseudo-code:

```tsx
<Entity>
  <DOMElement
    tag="input"
    attrs={{
      type: "search",
      placeholder: "Search docs",
      autocomplete: "off",
      spellcheck: false,
    }}
  />
  <Classes list={["search-input"]} />
  <SearchQuery value="" />
  <DOMBinding
    bind={binding(SearchQuery)}
    readEvent="input"
    read={el => (el as HTMLInputElement).value}
    write={(el, value) => {
      (el as HTMLInputElement).value = String(value ?? "");
    }}
  />
</Entity>
```

Then the search-specific system only handles filtering and result rendering.

Why this matters:
- it is a good migration test case
- if this still feels awkward after the refactor, the base design is probably too rigid

## How This Changes the View of `forms-ui`

Current view:
- `forms-ui` is a useful proof-of-concept vertical slice
- it already contains the right idea of descendant binding to root-owned state
- but it couples DOM directly to form fields

Refactor direction:
- design a general root-owned data binding foundation first
- then rebuild or refactor `forms-ui` on top of it

What `forms-ui` should eventually add above the binding foundation:
- validation
- dirty tracking
- touched state
- error rendering
- other form-specific semantics

Conceptually, the future layering is:

```text
DOMElement + DOMBinding
        |
  BindingTarget / binding(...)
        |
  root-owned data binding
        |
 forms-ui extras:
 - validation
 - dirty/touched
 - errors
 - form-specific helpers
```

## What We Have Rejected So Far

- making `DOMBinding` world/entity-imperative by default
  Why rejected:
  - too verbose for the common case
  - weakens optimization opportunities
  - muddies the base mental model

- forcing all binding through a generic `Value` intermediary
  Why rejected:
  - broad queries
  - extra marker/adaptor churn
  - awkward for existing domain components

- merging binding into `DOMElement`
  Why rejected:
  - improves local typing but hurts the architecture
  - many `DOMElement` entities need no binding at all

- generic `Root` marker as the primary lookup key for data binding
  Why rejected:
  - too ambiguous compared with nearest-ancestor-with-specific-component

## What We Have Not Fully Solved Yet

### Locked Core Decisions

The current binding-target foundation is now narrowed to the following rules.

#### 1. Binding Targets Are Descriptors Resolved at Runtime

- `binding(...)` creates a pure descriptor
- runtime systems resolve that descriptor for a specific `world` + binding entity
- the resolved target owns `get`, `set`, and `subscribe`

This keeps author-facing target construction separate from runtime owner lookup, subscriptions, and cleanup.

#### 2. Bindable Component Roots Use `{ value: ... }`

- bindable components expose their authoritative bindable root through a `value` field
- this should be TypeScript-enforced by the binding APIs
- component-backed bindings start from `component.value`
- derived bindings are then relative to the current target value

Why:
- one uniform mental model
- easier type inference
- simpler path composition for same-entity and ancestor-owned data

#### 3. Owner Resolution Is Nearest Matching Component, Including Self

- `binding(Component)` means “walk upward to the nearest entity with this component, including the current entity”
- resolved owners are cached
- cached owners must be re-resolved when scope-defining structure changes

Scope-defining changes include:
- ancestry changes
- add/remove of the target component on the binding entity or an ancestor in the active scope

#### 4. Subscription Is Required, and Broad Is Acceptable

- every resolved target must implement `subscribe`
- `subscribe` means “notify whenever the bound value may have changed”
- targets may over-notify
- plain component/path bindings subscribe at the owner-component level by default
- exact-path subscription remains an optimization, not a foundational requirement

This is a likely hot path. Revisit performance after the rest of the design is implemented.

#### 5. Missing Structure Is Strict by Default

- path traversal is represented as ordered, type-safe path steps
- missing structural intermediates throw descriptive errors by default on both `get()` and `set()`
- bindings do not implicitly create missing structure

This is intentionally strict so stale widgets and incorrect setup fail loudly instead of silently masking bugs.

#### 6. Plain Component Writes Use Mutation Semantics

- plain component-backed binding targets use mutation semantics, not immutable replacement, as the core model
- the preferred ECS primitive for this is a safe `world.mutate(...)`
- binding-target writes should use `world.mutate(...)` by default
- lower-level manual mutable access remains available for advanced/manual cases via `getMutable(...)` + `markMutated(...)` + `flush()`, but not as the default binding path

This matches the existing ECS design choice that components are mutable and avoids introducing a second write model just for bindings.

#### 7. Arrays Support Two Explicit Addressing Modes

Arrays are in scope, but the identity model is explicit:

```ts
binding(TripState).field("travellers").at(i)
binding(TripState).field("travellers").by("id", travellerId)
```

Rules:
- `.at(i)` is strictly positional
- `.by(key, value)` is semantic identity lookup within an array of records
- repeated widgets should generally prefer `.by(...)` when item identity matters
- normal JS array mutation such as `push`, `splice`, `sort`, and direct index assignment remains valid inside `world.mutate(...)`

We explicitly rejected:
- public `byRef(...)`
- content-hash identity
- implicit ref-or-index heuristics

Reference equality may still be used internally as an optimization, but it is not part of the public binding contract.

#### 8. No First-Class `Set` or `Map` Support in Core Bindings

- no first-class `Set` support in binding targets
- no first-class `Map` support in binding targets
- systems may still use `Set` and `Map` freely inside ECS/component state
- if consumers want ergonomic bindable collection semantics, prefer arrays, keyed arrays, or plain object fields

This keeps the core API smaller and avoids multiple competing collection mental models.

#### 9. Optional Values Are Explicit via `...Maybe()`

- strict bindings remain the default
- legitimate optional values are expressed explicitly with `...Maybe()` variants
- `Maybe` means the target value may be `undefined`
- `Maybe` does not imply silent creation of missing structure
- `Maybe` is for optional values, not for stale-widget recovery or structural absence that should change what gets rendered

Examples:

```ts
binding(ProfileState).fieldMaybe("nickname")
binding(ProfileState).field("profile").fieldMaybe("nickname")
```

Counter-example:
- `binding(EditorState).field("draft").field("title")` should still fail if `draft` is absent; the widget/binding likely should not exist in that state

#### 10. Core Bindings Stop at Addressing and Synchronization

The core layer is limited to:
- target description
- owner resolution
- path navigation
- read/write
- subscription

This means the core layer does not include:
- collection rendering helpers such as `.map(...)`
- conditional rendering helpers such as “if present render X else Y”

Those should be built afterward as higher-level sugar on top of the core target model.

### Remaining Design / Implementation Work

The foundation direction is now clear, but several follow-on tasks remain.

### Runtime Shape And ECS Support

We still need to define:
- the exact TypeScript shape of binding descriptors and resolved targets
- the exact `world.mutate(...)` API
- whether owner invalidation should build directly on current ECS mutation/event facilities or require a new primitive
- how resolved targets cache owner lookup and subscription cleanup

### Type-Safe Path Builder Details

We still need to define the exact typing/story for:
- `.field(...)`
- `.fieldMaybe(...)`
- `.at(...)`
- `.by(...)`

The goal remains strong TypeScript guidance and failure at compile time where possible.

### Existing `forms` Package Integration

This is still open, but intentionally not part of the core decision set yet.

We still need to decide:
- whether form instances expose or can be adapted to the binding-target model
- whether forms become one adapted component kind in the general binding system
- how much of touched/dirty/errors belongs above the shared foundation versus inside `forms-ui`

### Validation / Errors / Dirty / Touched

We intentionally postponed this to keep the base binding design clean.

Still open:
- what should live in the general root-owned binding layer?
- what should stay in `forms-ui`?
- how should error rendering bind to authoritative validation state?
- how should touched/dirty updates integrate with DOM bindings and widget-level behavior?

### Async

We have not designed async behavior yet.

This likely includes:
- async validation
- async option loading / widget data fetching
- async normalization or lookup flows
- what subscription/update model is needed when authoritative state changes from async sources

Possible pressure case:

```tsx
<AutocompleteBundle
  bind={binding(FormState).field("travellers").by("id", travellerId).field("country")}
/>
```

Questions:
- where does async option state live?
- should temporary loading/error state be local widget state?
- how should async authoritative updates propagate back through `DOMBinding`?

### Performance

We have identified performance as a design concern, but not finished the strategy.

Open questions:
- how aggressively should binding targets cache owner resolution?
- how granular should owner invalidation be?
- how much broad invalidation is acceptable for owner-level subscriptions?
- how much write-skipping and batching should be built into the base systems?
- what subscription/notification hot paths need optimization once the design is implemented?

## Current Working Direction

If implementation started immediately, the most likely order would be:

1. Keep `DOMElement` and `DOMBinding` separate.
2. Finalize the exact runtime/type shapes for binding descriptors, resolved targets, and `world.mutate(...)`.
3. Add static `DOMElement.attrs`.
4. Implement the first binding-target-based `DOMBinding` path against `{ value: ... }` components.
5. Validate the design against the docs search input.
6. Only after that, design higher-level sugar and refactor `forms-ui` onto the shared foundation rather than extending the current direct-coupling design.

### Trip Form with `DateRangeBundle`

The bundle version should differ only in ergonomics, not in data topology.

```tsx
<Entity>

  <DateRangeBundle

  />

```

Conceptual bundle implementation:

```tsx
function DateRangeBundle({ bind }) {
  return (
    <Entity>
      <DateRangeWidget />

      <Entity>
        <DOMElement tag="label" />
        <TextContent value="Start date" />
      </Entity>

      <Entity>
        <DOMElement tag="input" attrs={{ type: "date" }} />
        <DOMBinding
          bind={bind.field("start")}
          readEvent="input"
          read={el => (el as HTMLInputElement).value}
          write={(el, value) => {
            (el as HTMLInputElement).value = String(value ?? "");
          }}
        />
      </Entity>

      <Entity>
        <DOMElement tag="label" />
        <TextContent value="End date" />
      </Entity>

      <Entity>
        <DOMElement tag="input" attrs={{ type: "date" }} />
        <DOMBinding
          bind={bind.field("end")}
          readEvent="input"
          read={el => (el as HTMLInputElement).value}
          write={(el, value) => {
            (el as HTMLInputElement).value = String(value ?? "");

```

If the widget needs local transient state, it can add it separately:

<DateRangeUIState value={{ open: false, activeSegment: "start" }} />

That local state should not become the authoritative form value unless there is a compelling reason.

Nested arrays are explicitly in scope.

```tsx
function TravellerRow({ bind }) {
  return (
    <Entity>
      <Entity>
        <DOMElement tag="input" attrs={{ type: "text" }} />
        <DOMBinding
          bind={bind.field("name")}
          readEvent="input"
          read={el => (el as HTMLInputElement).value}
          write={(el, value) => {
            (el as HTMLInputElement).value = String(value ?? "");
          }}
        />
      </Entity>

      <Entity>
        <DOMElement tag="input" attrs={{ type: "number" }} />
        <DOMBinding
          bind={bind.field("age")}
          readEvent="input"
          read={el => Number((el as HTMLInputElement).value)}
          write={(el, value) => {
            (el as HTMLInputElement).value = String(value ?? 0);
          }}
        />
      </Entity>
    </Entity>
  );
}

<Entity>
  <FormState form={TripForm.create()} />
  {travellerIndexes.map(i => (
    <TravellerRow
      bind={binding(FormState, formAdapter).field("travellers").at(i)}
    />
  ))}
</Entity>
```

This example makes the remaining array questions concrete:
- how should row identity be stabilized?
- how should subscriptions behave when `removeAt(i)` shifts later items?
- should array mutation helpers belong on the binding target or elsewhere?
