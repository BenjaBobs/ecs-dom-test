# Design Choices

Architectural decisions for this ECS UI framework, including what we chose NOT to do and why.

## Context: UI vs Game ECS

This framework targets browser UI, not games. This context shapes every decision:

| Aspect           | Game ECS                       | UI ECS                          |
| ---------------- | ------------------------------ | ------------------------------- |
| Execution        | 60+ FPS game loop              | Event-driven                    |
| Entity count     | Thousands                      | Hundreds                        |
| Read:write ratio | Read-heavy (query every frame) | Balanced (query only on events) |
| Component churn  | Low (stable archetypes)        | High (dynamic state changes)    |
| Between events   | Systems still run              | Nothing happens                 |

## What We Do

### Reactive Systems (Group/Matcher-style)
Systems declare queries (`Entities.with([...]).without([...])`) and react via `onEnter`, `onUpdate`, and `onExit`.

**Why:** Event-driven UI means no wasted work between interactions. Queries stay simple and systems only run when membership changes or required components update.

### Component Index
`Map<componentTag, Set<EntityId>>` maintained on add/set/remove.

**Why:** O(1) lookup for "all entities with component X" without full archetype machinery. Good balance for UI's read:write ratio.

### Two-Pass Materialization
When materializing JSX, components/bundles are processed before child entities.

**Why:** Ensures parent's `DOMElement` exists before children try to attach to it. Deterministic DOM tree construction.

### Separate Reactive Systems for Behavior
`ClickableSystem`, `DisabledSystem`, etc. are separate from `DOMElementSystem`.

**Why:** Adding `Clickable` to an existing entity should work. Creation-time-only wiring is a bug, not a feature.

### `Classes` as Single Source of Truth
Removed `class` from `DOMElement`. All CSS classes go through `Classes` component.

**Why:** Two systems writing to `el.className` will fight. One source of truth eliminates the conflict.

### Value-Centric Data Binding
Bindability centers on components with a `{ value: ... }` shape. A generic `Value` component may still be useful as a catch-all, but the core binding model does not force all domain state through it. Domain-specific value components such as `SearchQuery` are valid bindable roots.

**Why:**
- Avoids tight coupling between DOM input state and form state
- Enables non-DOM entities to participate in data flows (e.g., pure data entities, future renderers)
- Preserves meaningful domain components such as `SearchQuery` instead of forcing adapter systems around a generic intermediary
- Supports multi-step transformations across boundaries (DOM ↔ value-shaped component ↔ Form) without mixing concerns

**Implemented core direction:**
- `DOMElement` stays a renderer primitive (no binding logic)
- `DOMElement` can carry static `attrs` for intrinsic element configuration, but not behavior or reactive binding logic
- bindable component roots use a uniform `{ value: ... }` contract
- `DOMBinding` is the base DOM bridge and targets a binding descriptor rather than a form field or arbitrary imperative world callback
- `DOMBinding` should separate read and write concerns explicitly:
  - `readEvent` drives DOM -> component value
  - `read` extracts a value from the element
  - `write` applies the current component value to the element
- Read-side controls such as debouncing belong on `DOMBinding`; write-side updates should remain immediate by default
- binding descriptors resolve against the nearest ancestor with the requested component, including self
- paths are explicit and type-oriented: `.field(...)`, `.fieldMaybe(...)`, `.at(i)`, `.by(key, value)`
- strict bindings are the default; `...Maybe()` is reserved for legitimate optional values
- plain component-backed writes should use `world.mutate(...)`; the lower-level manual path is `getMutable(...)` + `markMutated(...)` + `flush()`
- Form bindings should eventually adapt form fields onto the same binding-target model, rather than binding DOM elements directly to forms
- Thin ergonomic bundles such as `InputValue` or `CheckedValue` can sit on top of `DOMElement` + binding targets + `DOMBinding`, but are not required for the base design to work
- More imperative/custom bridges can exist separately when needed, but should not complicate the default `DOMBinding` path
- Avoid attaching `__entity` / `__world` properties to DOM nodes by default; current interaction systems already capture `world` and `entity` in listener closures, and world-owned element maps keep the renderer model cleaner. Revisit only if profiling shows lookup overhead is material.
- `DOMElement.attrs` should include static intrinsic element attributes/properties such as `type`, `placeholder`, `autocomplete`, `src`, `href`, `role`, `aria-*`, and `data-*`
- `DOMElement.attrs` should not absorb concerns already modeled elsewhere: `class`, `style`, text/HTML content, event handlers, or reactive value/checked synchronization

**Example:**
<div data-code-tabs>

```tsx
<Entity>
  <DOMElement
    tag="input"
    attrs={{ type: "search", placeholder: "Search docs", autocomplete: "off" }}
  />
  <SearchQuery value="" />
  <DOMBinding
    bind={binding(SearchQuery)}
    readEvent="input"
    read={(el) => (el as HTMLInputElement).value}
    write={(el, value) => {
      (el as HTMLInputElement).value = String(value ?? "");
    }}
  />
</Entity>
```

Non-JSX alternative:

```ts
world.createEntity(undefined, [
  DOMElement({
    tag: "input",
    attrs: { type: "search", placeholder: "Search docs", autocomplete: "off" },

    bind: binding(SearchQuery),
```

**Current low-level API shape:**

Binding builder:

```ts
const bind = binding(SearchQuery);
const nested = binding(TripState).field("range").field("start");
const optional = binding(ProfileState).fieldMaybe("nickname");
const row = binding(TripState).field("travellers").by("id", travellerId);
```

DOM bridge:

```ts
const DOMElement = defineComponent<{
  tag: keyof HTMLElementTagNameMap;
  attrs?: Record<string, string | number | boolean | null | undefined>;
}>("DOMElement");

const DOMBinding = defineComponent<{
  bind: BindingTarget<unknown>;
  readEvent?: keyof GlobalEventHandlersEventMap;
  read?: (el: Element) => unknown;
  write?: (el: Element, value: unknown) => void;
  readOnBind?: boolean;
  writeOnBind?: boolean;
  debounceMs?: number;
}>("DOMBinding");
```

Expected defaults:
- `writeOnBind: true`
- `readOnBind: false`
- `debounceMs: 0`
- if `readEvent` is omitted, the binding is write-only
- if `write` is omitted, the binding is read-only

Current behavior:
- `DOMBinding` requires `DOMElement`
- the bound owner is resolved by nearest matching component, including the current entity
- the DOM system remains responsible only for creating/removing elements
- the binding system is responsible for:
  - attaching the `readEvent` listener when configured
  - reading from the element and mutating the bound target value
  - writing the bound target value back to the element
  - skipping DOM writes when the DOM already reflects the current value
  - re-resolving cached owners when relevant ancestor/component mutations invalidate the current owner
  - cleaning up listeners/timers/subscriptions when the entity exits
- debounce only affects DOM -> state reads; state -> DOM writes stay immediate
- current first-pass support is focused on plain object/array paths under `{ value: ... }`

Current non-goals:
- cross-component TypeScript inference from `DOMElement.tag` into `DOMBinding.readEvent`
- arbitrary world/entity-imperative bridge callbacks in the base binding API
- first-class `Set` / `Map` binding support
- collection rendering helpers such as `.map(...)` on binding targets
- conditional rendering helpers in the core binding layer

Planned ergonomic helpers:

```ts
InputValue()
CheckedValue()
TextValue()
```

These should be thin bundles over `DOMElement` + binding targets + `DOMBinding`, with stronger defaults and less user-supplied boilerplate.

**Migration / reuse plan:**
- Reuse `getDOMElement(...)` and the world-owned entity -> element map from `@ecs-test/dom`
- Reuse the listener + subscribe loop shape already present in `@ecs-test/forms-ui`
- first migration target was the docs search input
- Later refactor:
  - move `@ecs-test/forms-ui` from direct DOM <-> form-field binding to the shared binding-target foundation
  - keep any higher-level form helpers as convenience layers over the decoupled primitives

### Feature-First Structure (Including Styles)
Features should own their components, systems, and styles together (folder per feature). Styles are applied via `Classes` and are kept in feature-scoped CSS files imported by that feature's entry module.

**Why:**
- ECS behavior and styling evolve together; co-locating avoids cross-folder churn
- Encourages reusable feature modules that can be moved into packages later
- Bundles can provide default `Classes` while `except={[Classes]}` lets consumers override

### Hand-Written Minimap Metadata + Generated SQLite Index
Agent-facing search should prefer high-signal human summaries over raw-code-only lookup. Source files can declare `@minimap summary` / `@minimap tags` metadata, and authored MDX pages can add `minimapSummary` / `minimapTags` in frontmatter. A generated SQLite index under `.minimap/` provides ranked lookup over that curated metadata, plus paths, exports, titles, and optional fallback text search. Generated API docs are excluded from the index by default so search results bias toward authored docs and source files. Rebuilds are written to a temporary SQLite file and swapped into place only after the new index passes an integrity check, so readers do not see a half-written database.

**Why:**
- keeps the source of truth in the files humans already edit
- avoids maintaining sidecar lookup files that drift from code/docs
- gives agents a fast local search surface without requiring a separate service
- preserves fallback retrieval from real file text when curated metadata is absent or incomplete

### World Externals (Escape Hatches)
Worlds can receive external dependencies (DOM factories, fetch, etc.) at construction time via `WorldExternals`. Packages extend the externals interface via module augmentation.

**Why:**
- Keeps non-ECS dependencies explicit and world-scoped
- Avoids hidden globals while remaining ergonomic in browser/runtime entry points
- Supports swapping dependencies for tests and non-DOM environments

**Example (conceptual):**
```ts
const world = new World({
  externals: {
    createElement: document.createElement.bind(document),
    rootContainer: document.getElementById("root") ?? undefined,
  },
});
```

### Runtime Entity (World-Owned Runtime State)
Each world has a single runtime entity for system-owned runtime state (e.g., DOM handler maps). Systems attach runtime-only components (like `DomRuntime`) to this entity.

**Why:**
- Keeps runtime wiring inside the world instead of in global WeakMaps
- Preserves ECS ownership without adding per-entity overhead
- Provides O(1) access to runtime state via a single entity

### World Isolation
Each `World` instance is fully independent. Multiple worlds can coexist in the same DOM (e.g., two app roots).

**Why:** Enables micro-frontends, isolated widgets, testing without cleanup.

**Implementation:** Runtime-only state is stored on the world's runtime entity (e.g., `DomRuntime`). External dependencies (like `createElement` and `rootContainer`) live in `WorldExternals`.

### ECS-DOM Cooperative Interaction Model
Todo-list and forms dogfooding exposed a recurring coordination issue:

- interaction often starts on one entity (child button/input)
- state often lives on a related entity (parent row/root widget)
- visuals often render on different related entities (siblings/children)

For browser UI, forcing every interaction through pure marker/system indirection creates extra boilerplate and can fight native semantics (input behavior, event flow, focus, form behavior).

**Decision direction:** keep ECS authoritative for state/composition/lifecycle, and use DOM-aligned handlers/bindings when they reduce unnecessary indirection.

**Current example:** `Clickable({ onClick })` short-circuits marker plumbing for local actions while keeping world mutations explicit.

**Solutions under consideration:**
1. Parent/child entity messaging
   - Typed, narrow APIs (`emitToParent`, `emitToChildren`) rather than a global event bus.
2. Widget-root state for multi-entity controls
   - One root entity owns widget data; child entities are interaction/render projections.
3. Event bubbling across ECS parent-child links
   - Implemented via typed entity events: `defineEvent`, `world.emit(...)`, `world.on(...)`.
   - Propagation model: handlers stop by default; call `event.bubble()` to continue upward.
4. Scoped processing ("tick specific entities")
   - Consider only after profiling because it may complicate ordering semantics.
5. Child-scoped query helpers (implemented)
   - Added `world.queryChildren(parent, ...components)` plus `world.findChild(...)` / `world.findChildMaybe(...)` to avoid manual `getChildren()` iteration and `[0]` lookups.
6. Binding primitives for forms/inputs
   - `InputValue`/binding-style components for DOM <-> ECS sync without repeated raw DOM casting.

**Near-term priority order:**
1. Event bubbling + payloads
2. Widget-root state pattern (forms first)
3. Child-scoped queries
4. Revisit scoped ticking only if measured bottlenecks remain

## What We Don't Do (and Why)

### No Archetypes / Chunk Storage
Archetypes group entities with identical component sets into contiguous memory tables.

**Why not:**
- Optimizes for iteration-heavy workloads (query every frame)
- We're event-driven - systems only run on mutations
- Component churn in UI (adding/removing state) would cause frequent archetype moves
- Added complexity not justified for our access patterns

### No Exposed Groups/Collectors API (Yet)
We do maintain per-query membership internally to drive `onEnter`/`onUpdate`/`onExit`, but we don't expose a public "group" API.

**Why:**
- Keeps the public surface area small while we stabilize query semantics
- Avoids committing to a group lifecycle/collector API too early

### No Frame-Based Scheduling
No `requestAnimationFrame` loop running systems continuously.

**Why not:**
- UI is event-driven, not frame-driven
- Running systems when nothing changed wastes CPU and battery
- Reactive queries naturally batch related changes within a single flush

### No Structural Sharing / Immutable Components
Components are mutable. No copy-on-write or persistent data structures.

**Why not:**
- Added complexity for time-travel debugging we don't need yet
- Mutable is simpler and faster for our scale
- Can revisit if debugging tools demand it

### No Built-in Async Scheduling
Async operations (fetch) are started in systems, with callbacks adding components and calling `flush()`.

**Why not build in async primitives:**
- Keeps the core synchronous and predictable
- Async is handled at the edges (systems that start fetches)
- `flush()` in callbacks is explicit - no magic

## Pending Decisions

### DOM Write Batching
**Status:** Not implemented yet

**Problem:** Multiple systems modifying DOM in one flush could cause layout thrashing.

**Options:**
1. Batch all DOM writes to end of flush (collect mutations, apply once)
2. Use `requestAnimationFrame` to defer DOM writes
3. Trust browser's natural batching within a single synchronous block

**Leaning toward:** Option 3 initially. Browsers batch synchronous DOM writes before paint. Only add complexity if we measure thrashing.

### Microtask Flush in Event Handlers
**Status:** Deferred

**Problem:** `world.flush()` in click handlers is re-entrant.

**Current:** Works fine for simple cases. Monitor for ordering bugs before adding complexity.

### Relationships Beyond Parent-Child
**Status:** Not implemented

**Potential:** First-class entity links like `(FocusTarget, entityId)`, `(FormOwner, entityId)`.

**Why interesting:** Queryable relationships could simplify focus management, form handling, tooltips.
