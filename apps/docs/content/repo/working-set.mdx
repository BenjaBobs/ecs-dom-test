## Current Plan: Binding System Exploration

Planning notes for `WS-R3-001` / `WS-R3-002`.

Detailed design exploration lives in [Current Todo](/repo/current-todo.html).

Goal:
- introduce a simple, flexible, low-boilerplate path for DOM <-> ECS value synchronization
- support composite widgets and root-owned state cleanly
- keep the default model explicit and teachable

Decisions so far:
- make `DOMBinding` the base DOM synchronization primitive
- use `readEvent` terminology rather than `event`
- support read-side timing controls such as debouncing on `DOMBinding`
- let `DOMElement` carry static intrinsic `attrs`, while keeping behavior/binding separate
- treat `forms-ui` as a proof-of-concept vertical slice rather than the final binding architecture
- explore binding targets and nearest-ancestor-with-specific-component lookup as the likely foundation for root-owned data binding
- do not add `__entity` / `__world` properties to DOM nodes by default; revisit only if profiling proves a meaningful lookup bottleneck

Phase 1: foundation
- extend `DOMElement` with static `attrs`
- add `DOMBinding`
- validate the simple text-input path first
- continue binding-target design until the root-owned data story is strong enough

Phase 2: ergonomics
- add thin helpers such as `InputValue`, `CheckedValue`, and possibly `TextValue`
- migrate at least one docs flow that currently reads raw DOM input state manually
- update guides/examples to show the new preferred pattern

Phase 3: forms decoupling
- refactor `forms-ui` away from direct DOM <-> form-field coupling
- introduce form/root-owned-data binding on top of the shared binding foundation
- keep higher-level forms conveniences as thin layers on top of the decoupled model

Existing code to reuse:
- `packages/dom/src/dom-element-systems.ts` for DOM element lifecycle and lookup
- `packages/forms-ui/src/systems.ts` for the current read/write/subscribe loop structure
- docs search and todo examples as early migration targets

Implementation checklist for `packages/dom`:
- add `packages/dom/src/features/value/components.ts`
  - define `Value`
  - keep the initial shape minimal: `{ value: unknown }`
  - define `DOMBinding`
  - make `DOMBinding` point at one explicit value-shaped component ref
  - add minimap metadata
- add `packages/dom/src/features/value/systems.ts`
  - implement `DOMBindingSystem`
  - require `DOMElement + configured value-shaped component + DOMBinding`
  - attach/remove `readEvent` listeners
  - support read-side debouncing
  - apply initial sync according to `readOnBind` / `writeOnBind`
  - write back only when the element state is actually different
  - clean up timers/listeners on exit
- update `packages/dom/src/dom-element-components.ts` and `packages/dom/src/dom-element-systems.ts`
  - add static `attrs` support to `DOMElement`
  - apply static intrinsic attrs/properties on mount/update
  - keep `class`, `style`, content, event handlers, and reactive value/checked sync out of `attrs`
- add `packages/dom/src/features/value/components.test.ts` or extend `packages/dom/src/dom.test.ts`
  - read-only binding
  - write-only binding
  - bidirectional text input binding
  - binding against a domain-specific value component (for example `SearchQuery`)
  - `writeOnBind` and `readOnBind` precedence
  - debounce behavior
  - listener cleanup on entity removal
- update `packages/dom/src/index.ts`
  - export `Value`, `DOMBinding`, and `DOMBindingSystem`
  - register `DOMBindingSystem` from `registerDOMSystems(...)`
- decide file placement for `DOMBinding`
  - likely colocate `Value` + `DOMBinding` under one new `features/value/` feature
  - avoid adding binding logic into `DOMElement` files
- keep v1 intentionally narrow
  - support text-like element value synchronization first
  - do not solve checked-state, text-content reflection, or form coupling in the first patch unless the implementation falls out naturally

First migration target:
- replace the manual search-input DOM event handling in `apps/docs/src/systems/search-system.ts` with `SearchQuery + DOMBinding`
- move static search input setup (`type`, `placeholder`, etc.) into `DOMElement.attrs`
- use that migration to validate whether the base API is actually understandable before adding any helper abstractions
